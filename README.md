Этот учебный проект представляет собой HTTP-сервер с игровой логикой, в которой игрок управляет предприятием «Угольная шахта». Цель игры — как можно быстрее вывести предприятие на стабильную прибыль и купить всё необходимое оборудование для полноценной работы. Проект задуман прежде всего как практическая отработка интерфейсов, конкурентного взаимодействия в Go, а также работы с HTTP и REST API, поэтому акцент здесь сделан не на идеальной архитектуре, а на закреплении ключевых тем курса.

В основе игры находится предприятие, которое с самого начала приносит пассивный доход: 1 уголь в секунду. Этот доход поступает на общий баланс предприятия и технически реализуется отдельной фоновой горутиной. Однако одного только пассивного дохода недостаточно, чтобы быстро накопить средства на дорогостоящее оборудование, поэтому игрок может нанимать шахтёров разных классов. Каждый шахтёр работает как отдельная горутина: через определённые интервалы времени он добывает уголь и передаёт его в общий поток прибыли предприятия. После того как у шахтёра заканчивается энергия, он завершает свою работу.

В игре предусмотрены три класса шахтёров. Маленький шахтёр стоит 5 угля, может выполнить 30 добыч, приносит по 1 углю каждые 3 секунды и не усиливается со временем. Нормальный шахтёр стоит 50 угля, рассчитан на 45 добыч, приносит по 3 угля каждые 2 секунды и также не меняет свои характеристики. Сильный шахтёр стоит 450 угля, работает 60 добыч, начинает с 10 угля за добычу и делает это каждую секунду, при этом после каждой новой добычи его эффективность увеличивается на 3 угля. Решение о том, какого шахтёра и в какой момент нанимать, принимает сам игрок.

Предприятие должно хранить данные как о текущем состоянии игры, так и о накопленной статистике. Оно знает, какие шахтёры работают в данный момент, к какому классу относится каждый из них и сколько энергии у них осталось. Дополнительно предприятие ведёт общую статистику по числу нанятых шахтёров каждого класса за всё время работы игры, включая тех, кто уже завершил работу. Все шахтёры хранятся через интерфейс Miner, чтобы в будущем можно было добавлять новые типы шахтёров без изменения внутренней логики предприятия.

Чтобы завершить игру, необходимо купить всё обязательное оборудование: кирки за 3000 угля, вентиляцию в шахту за 15000 угля и вагонетки за 50000 угля. После покупки всего оборудования игрок отправляет предприятию команду на завершение. В ответ предприятие должно корректно остановить всех работающих шахтёров через общий контекст, завершить HTTP-сервер и вернуть итоговую статистику, включая длительность прохождения, текущий баланс, информацию о нанятых шахтёрах и другие накопленные данные.

Взаимодействие с игрой осуществляется по HTTP с опорой на REST-подход. Через API должно быть возможно получать информацию о стоимости найма шахтёров по классам, нанимать новых шахтёров, запрашивать список всех активных работников и фильтровать его по классу. Аналогично для оборудования API должно позволять получать список доступных улучшений, узнавать их стоимость, покупать оборудование и видеть, что уже приобретено. Для самого предприятия должны быть доступны методы получения промежуточного состояния игры — например, текущего баланса и общей статистики — а также endpoint для корректного завершения игры.
